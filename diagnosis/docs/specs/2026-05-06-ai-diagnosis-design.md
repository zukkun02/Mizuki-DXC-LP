## 1. 概要

### 1.1 ゴール
**最終KPI**: jicoo 無料相談予約（CV）。
診断完了画面 → jicoo CTA で予約完了が成功状態。

### 1.2 補助KPI
- 診断完了率: 80%以上（5〜10分で完走できる設計のため）
- 結果LINEプッシュ受信率: 90%以上（LIFF経由の場合）
- 診断完了→jicoo クリック率: 30%以上（自社想定）

### 1.3 ポジショニング
診断シートは「中身の濃いリードマグネット」ではなく、**「相談前に感情を温める数字提示ツール」**。
ユーザーが自分で「月¥XX,XXX 削れる」を発見した瞬間が、jicoo クリックの最大確率タイミング。

### 1.4 旧仕様（Notion版・PDF版）との差分
本設計は `AI自動化診断シート概要.md` の HTML 化版だが、以下を **不採用** とする:
- ❌ Notion テンプレート版（不要）
- ❌ PDF版（不要）
- ❌ 30〜45分の業務棚卸し（5〜10分に圧縮）
- ❌ 10件以上の自由入力（プリセットチェック式に変更）
- ❌ メアド取得（不要、LINE連携で代替）
- ❌ Lステップ多段ナーチャリング（本設計対象外、別途）

## 3. 設問構成（全6問 + 時給確認）

### 3.1 画面フロー
- **1問1画面**（スマホ最適、集中度高い）
- **上部プログレスバー**: 「STEP 2/6」表示
- **下部固定**: 「次へ」ボタン（必須項目満たすまで非活性）+「戻る」リンク
- Q3（業務チェック）→ Q4（マトリクス）は **2画面に分割**（チェック画面 → 各選択業務の頻度時間入力画面）
- 「途中保存」UIは出さない（5〜10分で完走前提）。ただし内部的に localStorage に入力状態を都度保持し、ブラウザ閉じて再アクセスした場合は最後の入力位置から復帰可（**同一セッションの離脱復帰のみ**、過去診断の履歴一覧機能は §11.3 に従い v1.0 ではスコープ外）

### 3.2 設問詳細

#### Q1. 業種選択
- **形式**: カード単一選択（9枚）
- **選択肢**:
  1. 税理士・会計事務所
  2. 人材エージェント・採用支援
  3. コンテンツクリエイター・メディア
  4. 経営・戦略コンサル
  5. 整体院・治療院・サロン
  6. EC・物販
  7. コーチ・カウンセラー
  8. 不動産・建設
  9. **その他（業種を入力）** ← 業種名テキスト入力フィールドが下に出現
- **用途**: Q3 のプリセット業務リストの出し分け、結果画面のパーソナライズ冒頭文

#### Q2. 事業規模
- **形式**: 単一選択
- **選択肢**:
  - 1人（個人事業主・フリーランス）
  - 2〜5人（零細）
  - 6〜15人（小規模）
  - 16〜30人（中規模）
  - 30人以上
- **用途**: 規模別デフォルト時給の決定（後段で変更可）、結果メッセージのトーン調整

#### Q3. あるある業務チェック
- **形式**: 業種別プリセット業務リスト 12〜15個から **複数選択**（タップで✓トグル）
- **下部**: 「その他の業務（任意・自由記入）」テキスト枠 1つ
  - 自由記入された業務は **ROI計算からは除外**、結果画面に「気になる業務」として再掲のみ
- **必須バリデーション**: プリセットから **1件以上のチェック必須**。自由記入のみでは「次へ」非活性（ROI計算が成立しないため）
- **業種別プリセット**: §4 参照

#### Q4. 各業務の頻度・所要時間
- **形式**: Q3 でチェックした業務をすべて1画面に並べ、各業務に対し **頻度ドロップダウン × 時間ドロップダウン** をタップで設定
- **頻度選択肢**: 毎日 / 週次 / 隔週 / 月次 / 不定期
- **時間選択肢**: 5分以下 / 15分 / 30分 / 60分 / 2時間 / 3時間以上
- **デフォルト**: なし（明示選択させる）
- **画面**: 業務名がそのまま縦に並び、右側に2つのドロップダウン。スクロール可。
- **用途**: 月時間集計、ROI計算

#### Q5. 一番取り戻したいもの
- **形式**: 単一選択
- **選択肢**:
  - 時間そのもの（残業/休日業務を減らしたい）
  - ミスの削減（人為的なミスをなくしたい）
  - 属人化の解消（特定の人に依存しない仕組みを作りたい）
  - 売上拡大（人を増やさず売上を伸ばしたい）
  - 採用コスト削減（人を雇わずに業務を回したい）
- **用途**: 結果画面の冒頭メッセージとTOP3業務の推奨アクション文の出し分け

#### Q6. Notion・AI 利用状況
- **形式**: 単一選択
- **選択肢**:
  - 未経験（Notion を使ったことがない）
  - 触ったことがある（メモやタスクで使った程度）
  - 業務で使っている（チームで運用中）
  - 使いこなしている（複雑なDB設計やAI連携も）
- **用途**: 結果画面の推奨アクション語り口の出し分け
  - 未経験 → 「Layer 1（Notion設計）から段階的に」
  - 触った/使ってる → 「Layer 2（AI自動化）が最も効きます」
  - 使いこなしてる → 「Layer 3 エージェント設計が次のフロンティア」

#### Q7. 時給単価の確認（任意調整）
- **形式**: 結果計算前にワンクッション、規模別デフォルト値を表示してユーザーが変更可能
- **規模別デフォルト時給**:
  - 1人 → ¥3,500（個人事業主の時間機会損失）
  - 2〜5人 → ¥2,800（社保込み正社員平均）
  - 6〜15人 → ¥3,200（マネジャー混在）
  - 16〜30人 → ¥3,500
  - 30人以上 → ¥3,800
- **UI**: 「あなたの時給目安は ¥2,800 です（社保込み正社員平均）」と表示、その下に修正用の数値入力欄
- **下部脚注**: 「迷ったら標準値のままでOK。出典: 厚労省賃金構造基本統計調査・中小企業実態調査より算出」
- **用途**: ROI計算の信憑性UP、テンプレ感の払拭

## 5. 計算ロジック

### 5.1 月時間集計

各選択業務 `i` について:

```
月時間_i = 1回所要時間（分）_i × 月回数係数（頻度_i）÷ 60
```

月回数係数:
- 毎日 → 22
- 週次 → 4
- 隔週 → 2
- 月次 → 1
- 不定期 → 1（推測）

**1回所要時間の数値変換**: 5分以下=5 / 15分=15 / 30分=30 / 60分=60 / 2時間=120 / 3時間以上=180

### 5.2 集計指標

| 指標 | 式 |
|---|---|
| 総業務時間（月） | Σ 月時間_i |
| 自動化候補時間（月） | Σ {月時間_i \| recommended_layer ≠ 要相談} |
| 削減見込み時間（月） | 自動化候補時間 × 0.66 |
| 削減見込み人件費（月） | 削減時間 × 時給（Q7で確定した値） |
| 年間削減効果 | 月削減人件費 × 12 |
| Build Standard 投資回収期間 | ¥450,000 ÷ 月削減人件費 |

### 5.3 4層マッピング件数

各 Layer に該当する業務件数を集計:
- Layer 1+2: `recommended_layer ∈ {Layer 1+2}`
- Layer 2: `recommended_layer ∈ {Layer 2}`
- Layer 2+3: `recommended_layer ∈ {Layer 2+3}`
- Layer 3 / 要相談: それ以外

### 5.4 最優先業務 TOP3 抽出

ソート条件:
1. 月時間_i 降順
2. recommended_layer の自動化容易度（Layer 1+2 > Layer 2 > Layer 2+3）

上位3件を表示。各業務に推奨アクション文を併記:
- Layer 1+2: 「Notion DB化 + AI連携で削減可能」
- Layer 2: 「AI自動化（GPT/Claude等）で対応可能」
- Layer 2+3: 「自動化＋エージェントで一次対応無人化」
- Layer 3: 「複雑なエージェント設計で対応、要相談」

### 5.5 計算前提の根拠（脚注として表示）

- **削減係数 0.66**: 自動化候補のうち実際に削減可能な保守的見積もり（実績ベース）
- **月稼働日 22**: 平日換算（土日除く・祝日含まず）
- **Build Standard 中央値 ¥450,000**: LP料金プラン（¥300〜600k）の中央値
- **時給規模別デフォルト**: 厚労省賃金構造基本統計調査・中小企業実態調査より算出（社保込み）

### 6.2 パーソナライズ仕掛け（テンプレ感の徹底排除）

| 要素 | 動的化方法 |
|---|---|
| 冒頭文の業種・規模 | Q1（業種または自由入力業種名）+ Q2（人数）をエコーバック |
| 業務名表示 | Q3 で選択した業務のみ表示（プリセット全部ではない） |
| 月時間 | Q4 の入力ベース |
| 4層マッピング件数 | 選択業務の `recommended_layer` から動的算出 |
| TOP3 業務 | 月時間 × 自動化容易度で動的抽出 |

## 7. LIFF + Push 仕様

### 7.1 構成

```
┌──────────────────────────────┐
│ 診断HTML（LIFF アプリ）        │
│ ───────────────────────────  │
│ ・LIFF SDK 初期化              │
│ ・liff.getProfile() で userId │
│ ・診断完了時に POST /api/push │
└──────────────┬──────────────┘
               │ HTTPS POST
               ↓
┌──────────────────────────────┐
│ Cloudflare Workers           │
│ ───────────────────────────  │
│ POST /api/push               │
│ - userId, 結果サマリ          │
│ - LINE Channel Access Token  │
│   で /v2/bot/message/push 叩く │
└──────────────┬──────────────┘
               │
               ↓
┌──────────────────────────────┐
│ LINE Messaging API           │
│ → ユーザーのトーク画面に通知  │
└──────────────────────────────┘
```

### 7.2 必要な準備

- [ ] LINE Developers コンソールで Messaging API チャネル作成（既存LINE公式アカウントと別チャネルになる場合あり、要確認）
- [ ] Channel Access Token を取得
- [ ] LIFF アプリを登録、Endpoint URL に診断HTMLのURLを設定
- [ ] Cloudflare アカウント取得（無料）
- [ ] Cloudflare Workers にデプロイ、環境変数に Channel Access Token 設定

### 7.3 Push メッセージ設計

LINE Flex Message を使い、リッチに見せる。構成案:

```
[ヘッダー] 業務分析結果
[本文]
  〇〇業のあなたの月削減ポテンシャル
  ¥XX,XXX
  年間 ¥X,XXX,XXX 相当
  最優先業務 TOP3:
   1. 〇〇
   2. △△
   3. □□
[フッター] [ 無料相談を予約する ] ボタン → jicoo URL
```

### 7.4 セキュリティ

- userId は LIFF SDK 経由でフロントが取得 → POST で送信
- Cloudflare Workers では受け取った userId をそのまま LINE Messaging API に渡す
- Channel Access Token は Workers 環境変数のみに保持（フロント露出禁止）
- CORS: `https://liff.line.me`（LIFF）と本HTMLのオリジンのみ許可

### 7.5 エラー時の挙動

- Push 失敗（API Error / token切れ）: 画面表示は通常通り出す、Push送信失敗は内部ログに残すのみ
- LIFF 外アクセス: Push スキップ、Push 完了表示も非表示
- ネットワーク切断: localStorage に結果を保存しているので、画面表示は維持

### 9.1 単一HTMLか分割か

- **単一HTML（推奨）**: index.html に全部入れ、Tailwind CDN でスタイル、JSもインライン
- 理由: LP と同じ単一ファイル方式で運用が楽。診断HTMLの規模なら 1500行以下に収まる
- データ（業種別プリセット業務）はJSオブジェクトで埋め込み

### 9.2 ホスティング

- 本HTMLはどこにデプロイするか（実装フェーズで決める）:
  - 候補1: 既存LP と同じホスティング（Vercel / Netlify / GitHub Pages 等）の `/diagnosis/` パス
  - 候補2: 独立ドメイン or サブドメイン（`diagnosis.dxpartners.jp` 等）
  - 候補3: Cloudflare Pages（バックエンドWorkers と同居でデプロイ統一）
- LIFF アプリ Endpoint URL に登録する固定URLが必要
