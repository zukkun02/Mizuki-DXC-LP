# lp-implementer スキル

## 絶対にやらないこと(ハードガード)

- ❌ マーカー区間外の編集
- ❌ 既存 Tailwind class の書き換え
- ❌ 新規セクションの追加
- ❌ ステータスが「承認」以外のカードに手を出す
- ❌ 「自動実装可=✗」のカードに手を出す

これらを試みた場合は **必ず処理を中断し、ユーザーに警告**。

## 役割
承認済み改善提案カードを、安全にindex.htmlに反映する最終工程。
**Self-Refinement: 内部でlp-refinerを呼び、diffを推敲してから適用する。**

## 処理フロー

```
[1] 対象カード取得
    DB_LP改善提案から:
      ステータス = 承認
      自動実装可 = ✓
    の全カードを取得。優先度: 重要度High → Mid → Low の順。

[2] カードごとに以下をループ:

    [2-1] ドラフトdiff生成
          - 想定差分(Before/After)を読む
          - index.htmlの該当マーカー区間を特定
          - 必要なdiffを生成 → draft_diff

    [2-2] lp-refiner で推敲(Self-Refinement)
          artifact_type = "diff"
          draft = draft_diff
          rubric = (diff用ルーブリック: マーカー外侵入なし/HTML整合/class保持/視覚破壊なし)
          target_score = 85  ※実装はミスが許されないため通常より高めに設定
          max_rounds = 3

    [2-3] 採点が target_score 未満の場合
          → そのカードは「差し戻し」ステータスに変更
          → 推敲ログを「推敲ログ」列に書き戻して 次のカードへ
          (実装は強行しない)

    [2-4] バックアップ
          - index.html を index.html.bak にコピー

    [2-5] 適用
          - 推敲済みdiff を index.html に反映
          - 該当マーカー区間内のみ書き換え

    [2-6] 検証
          ✓ マーカー4ペアが全て残っているか
          ✓ <div> と </div> の総数が編集前と矛盾しないか
          ✓ 文字コードが UTF-8 のままか
          ✓ ファイルサイズが極端に増減していないか(±30%以内)

    [2-7] 検証失敗 → ロールバック
          - index.html.bak から復元
          - カードを「差し戻し」に変更、推敲ログに失敗理由を記録
          - 次のカードへ

[3] 全カード処理後、検証OKだった分だけまとめてコミット
    git add data/site.json index.html
    git commit -m "lp-implementer: 改善提案 N件を反映 (YYYY-MM-DD)"
    git push origin main
    → Vercelが自動デプロイ

[4] DB側を更新
    実装したカード:
      ステータス → 実装済
      差分プレビュー → git diff の要約(50行以内)
      リビジョン → 推敲した回数
      自己評価スコア → 最終スコア

[5] 完了報告
    ✅ 実装: N件
    ⚠ 差し戻し: M件 (理由を一覧で)
    📦 デプロイ: Vercelで自動反映中
```

## diff用ルーブリック(lp-refinerに渡す)

| 項目 | 配点 | 観点 |
|---|---|---|
| マーカー区間外への侵入なし | 25 | start/end の外を一切触らない |
| HTML閉じタグ整合 | 20 | <div>/<span>/<section>の数が編集前後で整合 |
| 既存class書き換えなし | 15 | Tailwindクラス名が変わっていない |
| 新セクション追加なし | 15 | 新規 <section> が増えていない |
| 視覚的破壊なし | 15 | 想定差分のBefore/Afterと一致 |
| アクセシビリティ後退なし | 10 | aria-label / alt属性が削除されていない |

## なぜ Self-Refinement を実装エージェントにも入れるか
- 提案カード時点で推敲済みでも、HTML diff化の段階で「マーカー区間外を触ってしまう」ような事故が起きうる
- 二重の防波堤として、適用前にもう一度ルーブリックで自己採点
- target_score=85 と 通常より高めに設定し、ミスが起きないように厳しめに

## 編集衝突の防止
- lp-sync と並列実行はしない(orchestratorが直列で呼ぶ)
- 1回の起動で複数カードを処理するときは、1カードずつ「適用→検証→次」の順で進める
- 全カード一括で書き換えてからまとめて検証はしない(失敗時に切り分け不能になるため)

## 失敗時の通知
カードが「差し戻し」になったら、Notionの該当カードに以下を書き込む:
- 推敲ログ: 何ラウンドで何が問題だったか
- ステータス: 差し戻し
- 自己評価スコア: 最終ラウンドのスコア

ユーザーは推敲ログを読んで、想定差分の書き方を直したり、自動実装可を ✗ にしてエージェントの手を離して手動で修正したりできる。
