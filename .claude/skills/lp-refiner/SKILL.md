# lp-refiner スキル(Self-Refinement レイヤー)

## なぜ必要か
LLM の一発出力は「もっともらしいが浅い」になりがち。
**自己批判 → 修正** を機械的に繰り返すことで、人間に渡す前に品質を引き上げる。
学術的には Self-Refine / Reflexion の系統。

## 役割
あらゆる「ドラフト + ルーブリック」を受け取り、改良された最終版を返す。
呼び出し側(レビュアー/実装者)は推敲のことを意識しなくてよい。

## 入力
```yaml
artifact_type: "proposal" | "diff" | "copy" | "review"
draft: <文字列(改善提案 / コードdiff / コピー文 / レビュー結果 など)>
rubric: <評価基準のリスト>
target_score: <0-100。これを超えたら採用。デフォルト 80>
max_rounds: <最大推敲回数。デフォルト 3>
context: <参考にしてよい補足情報(競合LP、index.htmlの該当箇所など)>
```

## 処理フロー(自己改善ループ)

```
ROUND 1
  ├─ Self-Critique:
  │   - rubric の各項目を 0-10 で採点
  │   - 合計スコアを算出(max 100)
  │   - 各項目で「なぜその点数か」「どう直せば上がるか」を一行で書く
  ├─ if スコア >= target_score:
  │   → 即採用してフローを抜ける
  └─ else:
      → 「最も低いスコア項目」を3つに絞り、修正方針を固める

ROUND 2
  ├─ Revise: ROUND1 の修正方針を反映して draft を書き直し → v2
  ├─ Self-Critique: 同じ rubric で再採点
  ├─ if スコア >= target_score: 採用
  └─ else: 次へ

ROUND 3 (最大)
  ├─ Revise: v3
  ├─ Self-Critique
  └─ 強制採用(これ以上回さない)

最後に推敲ログを構造化して返す:
  {
    final: <採用版>,
    score: <最終スコア>,
    rounds: <実施回数>,
    log: [
      { round: 1, score: 65, issues: [...], action: "修正方針" },
      { round: 2, score: 82, issues: [...], action: "..." }
    ]
  }
```

## ルーブリック例(参考)

### proposal(改善提案)用
| 項目 | 配点 | 観点 |
|---|---|---|
| 具体性 | 20 | Before/Afterが具体的で再現可能か |
| 根拠 | 20 | 「なぜ良くなるか」が説明できているか(競合参照・心理学・データ) |
| 実装可能性 | 15 | index.htmlのマーカー外を触らずに実装可能か |
| 影響度 | 15 | CV/視認性/動線にどれだけ効くか |
| 副作用 | 10 | 既存の動線・トーンを壊さないか |
| 重複排除 | 10 | 他カードと重複していないか |
| トーン整合 | 10 | 既存LPのです・ます調や落ち着いたトーンと合うか |

### copy(訴求100点)用
| 項目 | 配点 |
|---|---|
| 1秒で意味が掴めるか | 20 |
| ターゲット解像度 | 15 |
| 具体的ベネフィット | 15 |
| 数字/固有名詞 | 10 |
| 行動喚起の強さ | 10 |
| 既視感の少なさ | 10 |
| 文字数(ヒーロー≦25字 / CTAボタン≦12字) | 10 |
| トーン整合 | 10 |

### diff(実装エージェント用)
| 項目 | 配点 |
|---|---|
| マーカー区間外への侵入なし | 25 |
| HTML閉じタグ整合 | 20 |
| 既存class書き換えなし | 15 |
| 新セクション追加なし | 15 |
| 視覚的破壊なし | 15 |
| アクセシビリティ後退なし | 10 |

## 呼び出され方(他スキルからの利用)

各レビュアー/実装者の SKILL.md には次のように書く:

```
ステップ4. ドラフト生成: …
ステップ5. lp-refiner を呼ぶ:
  artifact_type = "proposal"
  draft = ステップ4の出力
  rubric = (proposal用ルーブリック)
  target_score = 80
  max_rounds = 3
ステップ6. lp-refiner の `final` をDBに保存
ステップ7. `score / rounds / log` をDBの「自己評価スコア」「リビジョン」「推敲ログ」に書き込む
```

## なぜスキル化するか
- 同じ推敲ロジックを4スキルに重複コピーするより、1ファイルにまとめる方が改善が早い
- ルーブリックを差し替えれば、コピー/提案/diff のどれでも適用できる
- 結果のフォーマット(score/rounds/log)が統一され、DBに整然と入る

## 注意
- 推敲は **コンテキスト消費が大きい**。max_rounds は 3 が上限
- 同じ修正を繰り返す「ループ事故」を検知したら強制終了(直前2ラウンドで rubric の上位3項目が同一なら停止)
- 採点の自己甘えを防ぐため、ROUND1 と ROUND2 では「なぜ点数を上げたか」を必ず log に書かせる
