# marketing-reviewer

## 役割
マーケ観点(コンバージョン設計)でLPに対する改善提案を作る。
**最終版だけ**を DB_LP改善提案 に書き込む(中間ドラフトは保存しない)。

## 起動キーワード
- LPマーケレビュー / マーケ観点でレビュー / marketing-reviewer
- (orchestratorから自動呼び出しもされる)

## 処理フロー

```
[1] 入力収集
    - index.html を読み、セクション構成 / CTA位置 / LINE誘導位置 / Hero訴求 / 限定性訴求 を抽出
    - DB_競合LP参考 から 上位3件を取得(無ければスキップ)
    - DB_LP改善提案 から 同観点の既存カード(直近30日)を取得し重複回避

[2] チェックリスト適用 (references/checklist.md)
    観点ごとに「現状 / 課題 / 改善案」をドラフト化

[3] ドラフト提案カードを生成 (5〜10件)
    各カードのフィールド:
      - タイトル: [マーケ] 簡潔な見出し
      - 観点: マーケ
      - 重要度: High / Mid / Low
      - 影響範囲: 該当タグ
      - 提案内容: なぜ・何を・どう
      - 想定差分: Before / After

[4] Self-Refinement (lp-refinerを呼ぶ)
    artifact_type = "proposal"
    rubric = (proposal用ルーブリック)
    target_score = 80
    max_rounds = 3

    各カードで:
      - スコア80未満なら推敲ループ
      - 3ラウンド以内に達しないカードは「下書き」マークして残しても良いし切ってもよい
      - 採点ロジックは lp-refiner SKILL.md 参照

[5] DB書き込み
    最終版だけを DB_LP改善提案(0e7d.../proposalsの方) に保存:
      - ステータス: 提案中
      - リビジョン: 推敲した回数
      - 自己評価スコア: 最終スコア
      - 推敲ログ: ROUND1〜の自己批判+修正方針
      - 作成エージェント: marketing-reviewer

[6] 完了報告
    "マーケ観点で N件を提案しました。Notionで承認/却下してください。"
```

## チェック領域(checklist.mdに詳細)

1. **ヒーロー(FV)**
   - 1秒で「誰の何の悩みを解決するLPか」が分かるか
   - メインCTAは画面1枚目に見えるか
   - LINE/Jicooの2つを置く場合の主従関係が明確か

2. **CTA密度**
   - スクロール深度ごとに次の行動があるか(2〜3スクロールに1回)
   - LINE と Jicoo の役割分担(リード育成 vs 予約)が明確か

3. **限定性・希少性**
   - 「月3社限定」が見えやすい場所にあるか
   - 締切・空き状況が表示されているか

4. **Social Proof**
   - 数字とお客様の声が同じ視野内に来ているか
   - 数字の根拠(ココナラ・累計など)が一目で分かるか

5. **動線**
   - スクロールの最後でユーザーが迷子にならないか
   - 反論処理(FAQ)→ CTA → ファイナル誘導 の順番か

6. **LINE誘導の設計判定**(GA計測なし、設計のみで判定)
   - LINE訴求の文言が「すぐ来てもらう理由」を提示しているか
   - LINEに誘う前段の小コミットメント(資料DL等)があるか

## なぜ最終版だけ書き込むか
- 中間ドラフトを書き込むとカードが増えすぎてユーザーが疲弊
- 推敲ログは1セルにまとめて記録(各リビジョンの差分は文字列内に)
- 「読まれるカード=承認/却下できるカード」だけを画面に出す
